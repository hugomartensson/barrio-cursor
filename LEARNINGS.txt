================================================================================
LEARNINGS FROM BARRIO — Reusable Principles for Future Projects
================================================================================

These learnings are derived from building the Barrio hyperlocal events app.
Each section states a general principle, illustrates it with Barrio examples,
and suggests how to apply it in future projects. Use guidelines.txt for
project-specific rules and checklists; use this file for portable wisdom.

================================================================================
1. DEPENDENCY & ENVIRONMENT VALIDATION
================================================================================

**Principle:** Never assume dependencies are installed or services are running.
Verify explicitly before starting work that requires them.

**Barrio example:**
- Missing Docker/PostgreSQL caused hours of debugging "connection refused" before we
  added a check-deps script and fail-fast startup checks.
- We run `npm run check-deps` (and health checks) before dev or tests.

**Apply elsewhere:**
- Add a single command (e.g. `check-deps` or `preflight`) that validates: runtime
  (Node/Python/etc.), database, env vars, optional services.
- Server startup: fail fast with clear messages if DB or required env is missing.
- Document "run X first" in README and optionally in dev scripts.

================================================================================
2. AI-GENERATED CODE — VERIFY BEFORE TRUSTING
================================================================================

**Principle:** Do not trust AI-generated code without build/test verification.
Especially for framework APIs: AI often invents plausible-looking properties or
methods that do not exist.

**Barrio example:**
- PhotosPickerItem.identifier → does not exist; real options include
  itemIdentifier or supportedContentTypes.
- Same for .loadImage() → use loadTransferable(type: Image.self).
- We added a mandatory checklist (Xcode build, autocomplete check) to guidelines
  and avoided many similar mistakes.

**Apply elsewhere:**
- For any language: run compiler/linter/tests before committing.
- For Swift: use Xcode autocomplete as source of truth; if a property/method
  doesn’t appear, don’t use it.
- When AI suggests framework APIs: verify in official docs, then build. Never
  paste-and-commit without verification.

================================================================================
3. SWIFT / SWIFTUI — COMMON PITFALLS
================================================================================

**Principle:** Swift and SwiftUI have sharp edges (type-checking, actor
isolation, optionality). Small, verified changes and consistent patterns
reduce debug time.

**Barrio example:**
- Huge `body` blocks caused "unable to type-check this expression in reasonable
  time". Fix: split into `@ViewBuilder` computed properties (e.g. mainContent,
  toolbarContent).
- `onChange(of: CLLocationCoordinate2D?)` failed until we added
  `Equatable` conformance for `CLLocationCoordinate2D` in a shared types file.
- Mutating `@Published` from async closures → "main actor-isolated property cannot
  be mutated from Sendable closure". Fix: wrap updates in `Task { @MainActor in ... }`.
- `catch { error = ... }` fails (error immutable); use `catch let err { ... }`.
- ViewModels often use `errorMessage: String?` rather than `error: Error?`;
  we learned to grep the ViewModel before using supposed property names.

**Apply elsewhere:**
- Keep SwiftUI `body` small; extract subviews or computed properties.
- Define shared types (enums, coordinate types) in one place to avoid
  "ambiguous" / "invalid redeclaration" errors.
- Always verify property names and signatures in the actual type definition
  before using them in views.

================================================================================
4. UPLOADS & MEDIA — PREFER DIRECT CLIENT UPLOADS
================================================================================

**Principle:** For user-generated media, direct client uploads to storage
(S3, Supabase, etc.) via signed URLs reduce server load and improve
latency. Server validates and issues URLs; client uploads binary.

**Barrio example:**
- We moved from server-proxied uploads (base64 → server → Supabase) to
  signed URLs. iOS uploads directly to Supabase. Upload time for ~12 MB
  video dropped significantly and server bandwidth use went down.
- We validate content type and video duration (e.g. max 15s) before
  creating signed URLs. Client also checks duration (e.g. AVAsset) pre-upload.

**Apply elsewhere:**
- Use presigned/signed URLs for large or frequent uploads.
- Server: validate metadata (type, size, duration) and optionally quotas;
  never stream full files through app servers if avoidable.
- Set upload timeouts and signed-URL expiry appropriately (e.g. 5–10 min
  for large files on slow networks).

================================================================================
5. NETWORK — LOCAL DEVICE TO HOST
================================================================================

**Principle:** Physical devices often cannot use localhost. Use the host’s
LAN IP, and be aware of WiFi client isolation (e.g. coffee shops, corporate
WiFi) blocking device–host communication.

**Barrio example:**
- Simulator: `127.0.0.1` worked; device failed until we used the Mac’s IP.
- Some WiFi networks block device–device traffic. We used Mac Personal
  Hotspot (iPhone joins hotspot) and wired backend to the hotspot IP
  (e.g. from `ifconfig bridge0` or `bridge100`).
- Server binds to `0.0.0.0` so it accepts connections from all interfaces.

**Apply elsewhere:**
- Document "Simulator: localhost / 127.0.0.1; Device: <host IP>" and how
  to obtain that IP.
- If "connection refused" or "cannot connect" on device, check WiFi
  isolation and suggest hotspot or USB tunneling.
- Explicitly bind server to `0.0.0.0` (or equivalent) for LAN access.

================================================================================
6. TROUBLESHOOTING LOOPS — PIVOT EARLY
================================================================================

**Principle:** If the same error persists after 2–3 serious attempts with the
same approach, assume a troubleshooting loop. Switch tools or methods instead
of repeating the same steps.

**Barrio example:**
- Prisma migrations repeatedly failed with auth/connection errors despite
  "correct" connection strings. After several retries, we used Supabase MCP
  (and Dashboard SQL) to apply changes directly. That bypassed CLI/auth
  issues and unblocked progress.

**Apply elsewhere:**
- Try fixes 2–3 times with clear variations; if no progress, stop.
- Consider: MCP or admin UIs, different auth (e.g. service role vs user),
  different connection paths (pooler vs direct, different ports).
- Ask: "Do we have access to [alternative tool] that could bypass this?"
- Document what worked and add it to runbooks or project learnings.

================================================================================
7. DATABASE — POOLED VS DIRECT CONNECTIONS
================================================================================

**Principle:** Use pooled connections for app traffic; use direct connections
for migrations, tests, or admin scripts when the pooler causes problems
(e.g. circuit breaker under many parallel connections).

**Barrio example:**
- App and dev use Supabase pooled URL (PgBouncer). Tests run in parallel
  and hit circuit breaker with the pooler. We override to `DIRECT_URL` in
  test setup (e.g. in Jest setup). Migrations use direct via Prisma
  `directUrl`. Pooled remains for normal runtime.

**Apply elsewhere:**
- Separate `DATABASE_URL` (pooled) and `DIRECT_URL` (or equivalent) in config.
- Run migrations and heavy parallel tests against direct connection.
- Document when to use which; ensure CI/test setup uses the right one.

================================================================================
8. GEOSPATIAL QUERIES — USE THE DATABASE
================================================================================

**Principle:** Do radius/distance filtering in the database with a proper
spatial index. Application-side distance math does not scale.

**Barrio example:**
- We use PostGIS, `ST_DWithin` on geography, and a GIST index on the
  location column. Nearby-event queries stay under ~100 ms. Filtering
  expired events (e.g. `endTime > NOW()`) is done in the same query.

**Apply elsewhere:**
- Use PostGIS (or equivalent) with a spatial index for "near me" / radius
  queries.
- Prefer `geography` for distance when relevant; keep indexing in mind.
- Combine spatial filters with other filters (e.g. time) in one query.

================================================================================
9. ERRORS — LOG FOR HUMANS AND TOOLS
================================================================================

**Principle:** Logs and API error payloads should support both humans and
automation (e.g. AI-assisted debugging). Include request IDs, context, and
structured fields so errors can be traced and copy-pasted into tools.

**Barrio example:**
- We use structured logs (e.g. Pino) with request IDs, endpoint, user ID,
  and sanitized context. API errors expose a machine-readable code, a
  user-facing message, and optional `details` (e.g. requestId, field).
- iOS logs format errors for easy copy-paste (URL, status, body, file,
  function) so we can quickly feed them into assistants.

**Apply elsewhere:**
- Add request IDs to all requests; include them in logs and error responses.
- Use consistent error codes and structure (e.g. code, message, details).
- Ensure logs are sufficient to debug without code inspection; avoid
  logging secrets or PII.

================================================================================
10. AUTHENTICATION & STORAGE
================================================================================

**Principle:** Use managed auth and storage where possible. Store tokens
securely (e.g. Keychain), not in UserDefaults or plain config.

**Barrio example:**
- Supabase Auth + Storage. JWTs in iOS Keychain; refresh on 401. User
  records synced to our DB after signup/login (find by email then id).
- We return 503 with diagnostic details when Supabase is unreachable
  instead of opaque 500s.

**Apply elsewhere:**
- Prefer platform-backed secret storage (Keychain, etc.) for tokens.
- Sync only what you need to your own DB; keep auth as source of truth.
- Differentiate "auth config wrong" vs "service down" in errors and logs.

================================================================================
11. TESTING & QUALITY GUARDRAILS
================================================================================

**Principle:** Automate checks that catch regressions early: build, lint,
tests. Use quality-control or verification scripts after major features
so "what was promised" matches "what was delivered."

**Barrio example:**
- Pre-commit: build, lint, test. After larger work we run
  `./quality-control/verify-after-work.sh` and review the generated report.
- We use test helpers (create user, cleanup, wait for DB) and a dedicated
  test DB setup. Direct DB URL for tests avoids pooler issues.

**Apply elsewhere:**
- Enforce build + lint + test before commit (hooks or CI).
- Add a "verify after feature" step (e.g. script + report) tied to
  requirements or PRD.
- Invest in test helpers and clear test DB strategy; avoid flakiness
  from shared or pooled resources.

================================================================================
SUMMARY — PRINCIPLES TO CARRY FORWARD
================================================================================

1. Validate dependencies and env before relying on them.
2. Verify AI-generated and framework code with build/tests and docs.
3. In Swift/SwiftUI: small views, shared types, correct actor/optional usage.
4. Prefer direct client uploads via signed URLs; validate on server.
5. For device–host dev: use host IP, consider WiFi isolation and hotspot.
6. Break troubleshooting loops by changing tools or approach after 2–3 tries.
7. Use pooled DB for app, direct for migrations and parallel tests.
8. Do geospatial work in the DB with proper indexes.
9. Log and structure errors for tracing and tool-assisted debugging.
10. Use secure token storage and managed auth/storage.
11. Automate quality checks and verify against requirements after big changes.

For Barrio-specific rules, checklists, and implementation details, see
guidelines.txt and PRD-TestFlight.md.
