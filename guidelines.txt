BARRIO - Development Guidelines
Product Requirements: See PRD-TestFlight.md (TestFlight Release Version)

This document provides development guidelines for the Barrio hyperlocal events app,
aligned with the PRD-TestFlight.md requirements. These guidelines ensure code quality,
security, and alignment with product requirements. For reusable principles and
learnings drawn from this project (useful in future work), see LEARNINGS.txt.

Workflow
- Initialize Git repo: git init
- Create GitHub repository for backup (free private repos available)
- Commit after each working feature with format: type(scope): description
  - Examples: feat(events): add creation endpoint, fix(auth): handle expired tokens
- Push to GitHub at end of each work session for backup
- Before committing: npm run build && npm run lint && npm run test must pass
- No feature branches needed for solo MVP; work directly on main
- Pre-commit hook: npm run build && npm run lint && npm run test
- Optional GitHub Actions on push to main (no branch protection needed for solo)
- Tag releases: git tag v0.1.0 when submitting to TestFlight

**Automated Quality Control (After Major Work):**
- After completing major features, run: `./quality-control/verify-after-work.sh`
- This verifies: What was promised → What was delivered, Completeness, Functionality
- Review `quality-control-report.txt` for issues
- Copy-paste error messages to AI assistant for diagnosis if needed
- Fix issues before continuing development
- See `quality-control/README.md` for details

AI Coding Tool Guidelines

**GOLDEN RULE: Never trust AI-generated code without verification. Always build/test before accepting.**

- Review all generated code before committing; AI hallucinates APIs and creates security holes.
- Watch for:
  - SQL injection (string concatenation in queries)
  - Missing input validation
  - Hardcoded secrets
  - Console.log instead of proper logging
  - Missing error handling
  - N+1 query patterns
  - Non-parameterized database queries
  - **Swift/SwiftUI API misuse (AI's #1 mistake: inventing properties/methods that don't exist)**
  - Deprecated iOS APIs (e.g., CLGeocoder in iOS 26.0+ should use MapKit)
  - Incorrect SwiftUI closure signatures (e.g., .onChange API changes in iOS 17+)

**Common Mistakes to Avoid:**

See PRD-TestFlight.md for feature requirements. Common implementation mistakes:

❌ **Don't:** Create separate endpoints for like/going/interested  
✅ **Do:** Single `/api/events/:id/interested` endpoint (see PRD Section 5.3)

❌ **Don't:** Store coordinates as primary, address as optional  
✅ **Do:** Address is PRIMARY field, coordinates derived via reverse geocoding (see PRD Section 4.1)

❌ **Don't:** Allow editing past events  
✅ **Do:** Enforce edit permissions - only future/ongoing events editable (see PRD Section 4.2)

❌ **Don't:** Let users create unlimited plans  
✅ **Do:** Enforce 5 plan limit with friendly error message (see PRD Section 7.1)

❌ **Don't:** Use "Like" or "Going" terminology anywhere  
✅ **Do:** Use "Interested" terminology throughout (see PRD Section 5.3)

❌ **Don't:** Allow users to follow themselves  
✅ **Do:** Validate `followerId !== followingId` (see PRD Section 6.1)

❌ **Don't:** Show private account events to non-followers  
✅ **Do:** Filter events by follow relationship for private accounts (see PRD Section 6.1)

❌ **Don't:** Auto-advance story viewer media  
✅ **Do:** Manual swipe only (see PRD Section 2.3)

❌ **Don't:** Implement advanced text overlay features  
✅ **Do:** Basic features only (see PRD Section 4.1)

❌ **Don't:** Use push notifications for follow requests  
✅ **Do:** In-app notifications only (see PRD Section 6.1)

**iOS/Swift Type & Function Organization:**

❌ **Don't:** Define the same enum/struct/function multiple times (causes "ambiguous" or "invalid redeclaration" errors)
✅ **Do:** Define shared types in `Models/SharedTypes.swift`, keep functions in appropriate files

**SwiftUI Complexity Management:**

❌ **Don't:** Create massive `body` properties with hundreds of lines (causes "unable to type-check this expression in reasonable time" errors)
✅ **Do:** Break up complex views using `@ViewBuilder` computed properties:
   - Extract main content: `private var mainContent: some View { ... }`
   - Extract toolbars: `@ToolbarContentBuilder private var toolbarContent: some ToolbarContent { ... }`
   - Extract overlays: `@ViewBuilder private var loadingOverlay: some View { ... }`
   - Keep `body` simple: `var body: some View { NavigationStack { mainContent.toolbar { toolbarContent } } }`

**SwiftUI Type Requirements:**

❌ **Don't:** Use `.onChange(of: CLLocationCoordinate2D?)` without Equatable conformance
✅ **Do:** Add `extension CLLocationCoordinate2D: Equatable` in `Models/SharedTypes.swift`:
   ```swift
   extension CLLocationCoordinate2D: Equatable {
       public static func == (lhs: CLLocationCoordinate2D, rhs: CLLocationCoordinate2D) -> Bool {
           lhs.latitude == rhs.latitude && lhs.longitude == rhs.longitude
       }
   }
   ```

**ViewModel Property Verification:**

❌ **Don't:** Assume property names without checking ViewModel definition
✅ **Do:** Always verify actual property names in ViewModel class before using:
   - Check: `grep "var error\|@Published.*error" ViewModel.swift`
   - Common: `errorMessage: String?` not `error: Error?`
   - Wrap String errors in NSError for ErrorView: `NSError(domain: "...", code: -1, userInfo: [NSLocalizedDescriptionKey: errorMessage])`

**Error Handling Pattern:**

❌ **Don't:** Use `catch { error = ... }` (error is immutable)
✅ **Do:** Always use `catch let err { error = err }` pattern for proper error handling

**Type Definition Rules:**
- **Shared enums/types** (used by 2+ views): `Models/SharedTypes.swift`
  - Examples: `TimeFilter`, `SortOption`, shared filter enums
- **API response types**: `Services/APIService.swift` (with the API methods that use them)
  - Examples: `UserProfileResponse`, `EventResponse`, `PlansResponse`
- **Domain models**: `Models/` directory
  - Examples: `Event`, `User`, `Plan` (core domain entities)
- **View-specific types**: In the view file (only if used by that view alone)
  - Examples: Local helper structs, preview data

**Function Definition Rules:**
- **API methods**: `Services/APIService.swift` (check for existing similar methods before adding)
- **View helpers**: In view file or separate helper file
- **Before adding a function**: Search for existing similar functions to avoid duplicates
  ```bash
  grep -r "func functionName" ios/
  ```

**Before accepting AI suggestions - REQUIRED CHECKLIST:**

1. **For Swift/iOS code:**
   - ✅ Does it compile in Xcode? (⌘B - must succeed)
   - ✅ Does Xcode autocomplete show the properties/methods? (If not → they don't exist)
   - ✅ Are there compilation errors? (If yes → fix before committing)
   - ✅ Verified API usage against Apple documentation?
   - ✅ **No duplicate definitions?** (Check for "ambiguous" or "invalid redeclaration" errors)
   - ✅ **Shared types in SharedTypes.swift?** (If used by multiple views)
   - ✅ **No duplicate functions?** (Search before adding new API methods)
   
2. **For all code:**
   - ✅ Does it pass linting?
   - ✅ Are there tests?
   - ✅ Does it match the complexity limits?
   - ✅ Is error handling present?

3. **When AI suggests framework APIs:**
   - ✅ STOP - Don't copy/paste immediately
   - ✅ Verify property/method name in official documentation
   - ✅ Test autocomplete in Xcode (for Swift) or IDE (for other languages)
   - ✅ Build and test before accepting
   
- When AI suggests refactoring working code: push back unless there's a clear bug or performance issue.

**Common pattern of AI mistakes:**
1. AI suggests: `item.identifier` (seems logical)
2. Reality: `PhotosPickerItem` doesn't have `.identifier` 
3. Result: Compilation error
4. Prevention: Always verify in Xcode/Apple docs before using

**Avoiding Troubleshooting Loops - When to Use Alternative Approaches**

**CRITICAL: If the same error persists after 2-3 attempts, consider alternative tools/approaches instead of repeating the same troubleshooting steps.**

**Pattern to Recognize:**
- Same error message appears multiple times (e.g., "authentication failed", "connection refused")
- Same troubleshooting steps attempted repeatedly (e.g., resetting password, checking connection strings)
- No progress after 3+ attempts with the same approach

**When This Happens:**
1. **Stop and assess:** Are we in a troubleshooting loop?
2. **Consider alternative tools:**
   - MCP servers (e.g., Supabase MCP for direct database access)
   - Admin interfaces (e.g., Supabase Dashboard SQL Editor)
   - Alternative authentication methods (service role keys, OAuth, etc.)
   - Different connection methods (pooler vs direct, different ports)
3. **Ask the user:** "Do you have access to [alternative tool] that might bypass this issue?"
4. **Document the solution:** Once found, add to LEARNINGS.txt

**Example (Database Connection Issues):**
- ❌ **Don't:** Keep resetting passwords and trying connection strings 10+ times
- ✅ **Do:** After 2-3 failed attempts, ask: "Can you give me MCP access to Supabase so I can apply migrations directly?"

**Takeaway:** Alternative tools/interfaces often bypass authentication/connection issues that block standard CLI tools. Recognize loops early and pivot to better approaches.

iOS/Swift Specific Guidelines (PRD-TestFlight.md Section 2-5)

**Key Principles (see BUILD PREVENTION RULES below):**
- ✅ ALWAYS verify properties/functions exist before using them (use view tool)
- ✅ NEVER assume singleton pattern - check for `static let shared` first
- ✅ Work in small increments - ask user to build after each change
- ✅ Match types exactly in ternary expressions
- ✅ Use `catch let error` for error handling
- ✅ Verify property wrapper types (@EnvironmentObject vs @StateObject)

**CRITICAL: Always verify Swift/SwiftUI API usage against Apple documentation when AI suggests properties/methods**
- AI frequently hallucinates property/method names that don't exist (e.g., `PhotosPickerItem.identifier` → should be `itemIdentifier` or `supportedContentTypes`)
- Never trust AI suggestions for framework APIs without verification
- When AI suggests using a property/method, check Apple's official documentation first

**Build verification before committing:**
- Code MUST compile successfully in Xcode (⌘B) before committing
- NEVER commit code with compilation errors - fix errors first
- If Xcode shows compilation errors, do NOT accept "it should work" - the compiler is always right
- Run clean build (⇧⌘K then ⌘B) to catch hidden compilation issues

**API verification process:**
- Step 1: AI suggests API usage → **STOP**
- Step 2: Check Apple documentation (developer.apple.com/documentation)
- Step 3: Verify property/method exists in Xcode autocomplete (⌘Space after dot)
- Step 4: If Xcode autocomplete doesn't show it → property/method doesn't exist → don't use it
- Step 5: Build and test (⌘B) → must compile without errors

**Watch for API changes between iOS versions:**
- Project targets iOS 17+ (PRD Section 7: iOS 17+)
- Deprecated APIs: CLGeocoder → MapKit geocoding (iOS 26.0+)
- SwiftUI APIs evolve: `.onChange(of:)` closure signature varies by iOS version
- Check deprecation warnings in Xcode - don't ignore them

**Common AI hallucination patterns to watch for:**
- `PhotosPickerItem.identifier` → ❌ Doesn't exist (use `itemIdentifier` or `supportedContentTypes`)
- `PhotosPickerItem.loadImage()` → ❌ Doesn't exist (use `loadTransferable(type: Image.self)`)
- Properties that "make sense" but don't exist (e.g., `.identifier` on types that use `.itemIdentifier`)
- API methods with names that seem logical but aren't in the framework
- Always verify against Apple documentation or Xcode autocomplete

**When AI gives Swift code:**
- ✅ First: Copy to Xcode, try autocomplete on the API
- ✅ Second: Build (⌘B) - must compile without errors
- ✅ Third: Test the functionality
- ❌ NEVER: Assume AI code is correct without building/testing

**Xcode autocomplete is your friend:**
- If Xcode doesn't suggest a property/method after typing `.` → it doesn't exist
- Trust Xcode autocomplete over AI suggestions
- If autocomplete shows a different name → use that name, not what AI suggested

Code Quality and Style
- ESLint complexity rules are errors; pre-commit hooks block violations. Limits: cyclomatic 10; nesting 3; lines/file 300; lines/function 50; params 4; statements/function 20. See CODE_QUALITY.md for refactor patterns.
- ES modules only ("type": "module" in package.json).
- **TypeScript:** Use strict mode; prefer interfaces for object shapes, types for unions/intersections.
- **Swift:** Use structs for models, classes for ViewModels/services; prefer value types (structs) over reference types (classes) when possible.
- **Naming Conventions:**
  - Backend: camelCase for variables/functions, PascalCase for types/classes
  - iOS: camelCase for variables/functions, PascalCase for types/structs/classes
  - Database: snake_case for columns (Prisma maps automatically)
  - API endpoints: kebab-case (`/api/users/:id/follow`)

Logging (AI-Assistant Friendly)

**CRITICAL: All errors must be detailed enough for AI assistants to diagnose without code inspection.**

- Use structured logging (Pino recommended) with JSON output for easy parsing.
- Log levels: ERROR (unhandled failures), WARN (handled errors), INFO (key business events), DEBUG (development only).
- Always log: auth attempts (success/fail), event CRUD operations, media uploads, geospatial queries with params.
- Never log: passwords, full JWTs, user coordinates (log distance only).

**Structured Logging Format (Required for AI-Assistant Diagnosis):**
```typescript
logger.error({
  requestId: req.id,
  userId: authReq.user?.userId,
  endpoint: `${req.method} ${req.path}`,
  error: {
    name: error.name,
    message: error.message,
    code: error.code, // Custom error code if available
    stack: error.stack, // Include stack trace in logs (not in API response)
  },
  context: {
    // Include relevant context (eventId, planId, etc.)
    eventId: req.params.id,
    // Include request body/query (sanitized - no secrets)
    body: sanitizeRequestBody(req.body),
  },
  file: __filename, // File where error occurred
  function: 'functionName', // Function name where error occurred
}, 'Human-readable error description');
```

**Request ID Tracking (Required):**
- Every request must have a unique request ID (generated by middleware).
- Include request ID in all logs, error responses, and API responses.
- Format: `requestId: "req_abc123xyz"` (visible in logs and error responses).
- This allows tracing errors across server logs, iOS console, and API responses.

**Error Message Structure (Copy-Paste Friendly):**
- Errors must include: What failed, Why it failed, Where it failed, How to fix it.
- Format errors so they can be copy-pasted directly to AI assistant.
- Include file path, function name, and line number in logs (not in user-facing errors).

**Example Good Error Log:**
```
ERROR: Event creation failed
Request ID: req_abc123xyz
File: server/src/routes/events.ts:145
Function: createEvent
Error: ValidationError: Address is required
Context: { userId: "user_123", title: "Test Event", hasMedia: true }
User Message: "Please provide an event location"
Diagnostic: Check if reverse geocoding failed or address field missing
```

**Example Bad Error Log (Don't Do This):**
```
ERROR: Failed
```

Error Handling (AI-Assistant Friendly)

**CRITICAL: Errors must be detailed enough for non-coders to copy-paste to AI assistants.**

- Never use empty catch blocks; always log or re-throw with context.
- Custom error classes for: ValidationError, AuthError, NotFoundError, PermissionError, BusinessRuleError.
- Centralized error handler middleware catches all; formats per standard shape.
- Async functions: always use try/catch; no unhandled promise rejections.

**Error Response Format (API):**
```typescript
{
  error: {
    code: "VALIDATION_ERROR", // Machine-readable code
    message: "User-friendly message", // Human-readable, shown to user
    details: {
      // Optional: Detailed diagnostic info (for AI assistants)
      requestId: "req_abc123xyz",
      field: "address",
      reason: "Address reverse geocoding failed",
      suggestion: "Check if pin position is valid",
    }
  }
}
```

**Error Logging Format (Server Logs - Not User-Facing):**
```typescript
logger.error({
  requestId: req.id,
  error: {
    name: error.name,
    message: error.message,
    stack: error.stack, // Full stack trace in logs
  },
  context: {
    endpoint: `${req.method} ${req.path}`,
```

**iOS Error Handling (Copy-Paste Friendly):**

**CRITICAL: iOS errors must be formatted for easy copy-paste to AI assistants.**

- Log errors with full context: URL, HTTP status, response body, error domain/code.
- Include request ID from API response in iOS logs (always include if available).
- Format errors so they can be copy-pasted directly to AI assistant.
- Use multi-line string format for readability.
- Include: What failed, Why it failed, Where it failed (file/function), Request ID, Full error details.

**Required iOS Error Log Format:**
```swift
print("""
❌ [SERVICE_NAME]: [Operation] failed
   Request ID: [requestId from API response]
   Endpoint: [METHOD] [PATH]
   Status: [HTTP_STATUS_CODE] [STATUS_TEXT]
   Error Code: [ERROR_CODE from API]
   Message: [Error message from API]
   Details: [Additional diagnostic info]
   File: [FileName.swift]:[LineNumber]
   Function: [functionName]
   Context: { [relevant context like userId, eventId, etc.] }
   Full Response: [JSON response body if available]
""")
```

**Example iOS Error Log (Good):**
```swift
print("""
❌ APIService: Event creation failed
   Request ID: req_abc123xyz
   Endpoint: POST /api/events
   Status: 422 Unprocessable Entity
   Error Code: VALIDATION_ERROR
   Message: Address is required
   Details: Address reverse geocoding failed - pin position invalid
   File: APIService.swift:234
   Function: createEvent
   Context: { userId: "user_123", hasMedia: true, title: "Test Event" }
   Full Response: {"error":{"code":"VALIDATION_ERROR","message":"Address is required","details":{"requestId":"req_abc123xyz","field":"address"}}}

```

**Example iOS Error Log (Bad - Don't Do This):**
```swift

```

**iOS Error Handling Best Practices:**
- Always log request ID if available from API response.
- Include full API error response body in logs (for AI assistant diagnosis).
- Log both user-friendly message AND technical details.
- Use consistent error log format across all services (APIService, StorageService, etc.).
- Include file name and approximate line number in error logs.
- Never log sensitive data (passwords, tokens, user coordinates).

**Error Categories (Use Appropriate Code):**
- `VALIDATION_ERROR` (422): Input validation failed (include field name and reason)
- `AUTH_ERROR` (401/403): Authentication/authorization failed
- `NOT_FOUND` (404): Resource not found (include resource type and ID)
- `BUSINESS_RULE_ERROR` (400): Business rule violation (e.g., "5 plan limit reached")
- `SERVICE_UNAVAILABLE` (503): External service unreachable (include service name)
- `INTERNAL_ERROR` (500): Server-side error (include request ID for tracing)

**Best Practices:**
- Always include request ID in error responses and logs.
- Log full error details (stack trace, context) in server logs.
- Return user-friendly messages in API responses (no stack traces).
- Include diagnostic details in `error.details` for AI assistant diagnosis.
- Use consistent error codes across the application.
- Never log sensitive data (passwords, full JWTs, user coordinates).

Database and Migrations (PRD-TestFlight.md Alignment)

**Schema Requirements:**
- Use Prisma for migrations; commit all migrations and keep them idempotent.
- **User Model:** Must include `isPrivate` (boolean, default false), `followerCount`, `followingCount` (denormalized counts)
- **Event Model:** Must include `address` (string, PRIMARY field), `isFree` (boolean), `interestedCount` (renamed from `likesCount`)
- **MediaItem Model:** Must include `thumbnailUrl` (string | null) for videos
- **New Tables Required:** `Follow`, `FollowRequest`, `Interested` (replaces `Like`), `Plan`, `PlanEvent`
- **Tables to Remove:** `Like`, `Going` (not in PRD)

**Migration Strategy:**
1. Add new fields/tables first (non-breaking)
2. Migrate data: `Like` → `Interested` (copy all existing likes)
3. Update code to use new schema
4. Remove old tables (`Like`, `Going`) in separate migration after code is updated

**Geospatial Queries:**
- Use ST_DWithin with fixed 5 km radius (see PRD Section 2.1)
- GIST spatial index on location geometry column (already exists)
- All event queries must filter WHERE endTime > NOW() (or startTime > NOW() if endTime is null)
- Address is PRIMARY field; coordinates derived via reverse geocoding (see PRD Section 4.1)

**Background Jobs:**
- Daily cron job hard-deletes events where endTime < NOW() - 24 hours
- Same cron job archives plans when all events passed OR plan empty for 7+ days (see PRD Section 7.5)
- Document in deployment README

**Denormalized Counts Best Practice:**
- Maintain `followerCount`, `followingCount` on User (updated via triggers or application logic)
- Maintain `interestedCount` on Event (updated via triggers or application logic)
- Use transactions when updating counts to ensure consistency
- Consider database triggers for count updates (more reliable than application-level updates)

Transactions
- Wrap multi-step operations in transactions (e.g., create event + upload media references).
- Use connection pooling (pg-pool); max pool size = 20 for MVP.
- Always use parameterized queries; never string concatenation for SQL.

API and Validation (PRD-TestFlight.md Alignment)
- Routes use /api/ prefix. TestFlight ships as v1; breaking changes require v2.
- Validate all inputs at the edge (body/query/params schemas using Zod); reject with 4xx on invalid data.
- Standard error shape: { error: { code, message, details? } }; do not leak stack traces.
- Auth: email/password with verification; JWTs stored in iOS Keychain; token refresh endpoint required (PRD Section 7).
- Hard deletes only; no soft deletes.
- **Ownership verification patterns:**
  - DELETE /events/:id: Verify userId matches event creator
  - PATCH /events/:id: Verify userId matches event creator AND event is future/ongoing (see PRD Section 4.2)
  - DELETE /plans/:id: Verify userId matches plan owner
  - PATCH /plans/:id: Verify userId matches plan owner
- **Edit permissions:** Events can only be edited if startTime > NOW() OR (startTime <= NOW() < endTime) (see PRD Section 4.2).
- **Plan limit enforcement:** Check active plans count before creation; return friendly error if limit reached (see PRD Section 7.1).
- **Address validation:** Always reverse-geocode pin position to get address string; store address as PRIMARY field (see PRD Section 4.1).

Security (PRD-TestFlight.md Alignment)
- HTTPS only in production; reject HTTP.
- Rate limiting:
  - Auth endpoints: 5 requests/minute per IP
  - Event creation: 10/hour per user
  - Interested: 100/hour per user (see PRD Section 5.3 for terminology)
  - Follow actions: 50/hour per user
  - Plan creation: 20/hour per user
- Helmet.js for security headers.
- JWT: Store in iOS Keychain; short expiration (24h); refresh token pattern (see PRD Section 7).
- Input sanitization: trim whitespace, strip HTML tags from text fields.
- SQL injection prevention: parameterized queries only.
- CORS: whitelist client origin only; no wildcard in production.
- **Self-follow prevention:** Validate `followerId !== followingId` in follow endpoints (see PRD Section 6.1).
- **Ownership verification:** All edit/delete operations must verify userId matches resource owner.

Performance
- Database queries:
  - Single event lookup: <50ms
  - Nearby events query: <200ms
  - Add EXPLAIN ANALYZE to slow queries in tests
- API response times:
  - GET endpoints: <500ms p95
  - POST endpoints: <1s p95 (excluding media upload)
- Connection pooling required; no per-request DB connections.
- Pagination: limit=50 max for list endpoints.

Media Handling (see PRD-TestFlight.md Section 4.1)
- Media uploads use Supabase Storage presigned URLs only (direct client upload).

BUILD PREVENTION RULES (10 Rules):

**Rule 1: Verify .shared Singletons**
- Before using `ServiceName.shared`, check: `grep "static let shared" ios/path/to/ServiceName.swift`
- If NOT found → Use `@EnvironmentObject` or injected instance
- Example: `AuthManager` uses `@EnvironmentObject`, NOT `.shared`

**Rule 2: Verify Property Names**
- Use `view` tool to read service/model file before using properties
- Look for `@Published var propertyName` or `var propertyName`
- Common mistake: `authManager.user` → actual is `authManager.currentUser`

**Rule 3: Verify Function Signatures**
- Use `view` tool to read file containing function
- Find exact signature: `func functionName(param1: Type1, param2: Type2)`
- Provide ALL required parameters with correct names/types

**Rule 4: ViewModel Properties Must Exist**
- Use `view` tool to read ViewModel file
- Search for `@Published var propertyName`
- If NOT found → Add it first, THEN use it
- Common mistake: `viewModel.error` → actual is `viewModel.errorMessage`

**Rule 5: Small Increments - User Must Build**
- Make ONE small change
- Ask user: "Please build in Xcode (⌘B) to verify this compiles"
- WAIT for user confirmation before proceeding
- Fix errors before adding more code

**Rule 6: Match Types in Ternary Expressions**
- Both branches must return SAME type
- If types differ, wrap in `AnyShapeStyle()` or `AnyView()`
- Example: `Color` vs `LinearGradient` → wrap both in `AnyShapeStyle`

**Rule 7: Modern Swift Error Handling**
- Always use `catch let err` (not `catch { error = ... }`)
- Pattern: `catch let err { errorMessage = err.localizedDescription }`

**Rule 8: Verify Property Wrappers**
- `@EnvironmentObject`: Injected from parent/app level
- `@StateObject`: View OWNS the object (creates it)
- `@ObservedObject`: Passed from parent as parameter
- Use `view` tool to check how object is used elsewhere

**Rule 8.5: @EnvironmentObject Not Available in init()**
- Cannot access `@EnvironmentObject` in `init()`
- Initialize `@State` with defaults, use `.task` or `.onAppear` to load from `@EnvironmentObject`

**Rule 9: Read Error Messages Fully**
- Ask user for FULL error message with line numbers
- Use `view` tool to see actual code at that line
- Understand error before suggesting fix

**Rule 10: Verify Code Before Declaring Complete**
- Use `view` tool to re-read modified files
- Verify all properties/functions exist
- Check all parameters match function signatures
- THEN ask user to build (⌘B)

COMMON BUILD ERROR PATTERNS:

**Pattern 1: Duplicate Definitions**
- Error: `'TypeName' is ambiguous` or `Invalid redeclaration`
- Fix: Move shared types to `Models/SharedTypes.swift`
- Prevention: Search before adding: `grep -r "enum TypeName\|struct TypeName" ios/`

**Pattern 2: Missing Required Parameters**
- Error: `Missing argument for parameter 'paramName' in call`
- Fix: Use Xcode autocomplete to see all required parameters
- Prevention: Build frequently, search for all initializations when updating structs

**Pattern 3: Structural/Syntax Errors**
- Error: `Attribute 'private' can only be used in a non-local scope`
- Fix: Match braces carefully, ensure all code is inside struct
- Prevention: Use Xcode's brace matching (⌘⇧P), verify structure

**Pattern 4: Missing Import Statements**
- Error: `Cannot find type 'Combine' in scope`
- Fix: Add `import Combine` when using `@Published` or `ObservableObject`
- Prevention: Always add imports when using framework features

**Pattern 5: Name Conflicts**
- Error: `Invalid redeclaration of 'functionName()'`
- Fix: Rename function or property to avoid conflict
- Prevention: Search before adding: `grep -r "func functionName" ios/`

**Pattern 6: Variable Shadowing in Catch Blocks**
- Error: `Cannot assign to value: 'error' is immutable`
- Fix: Use `catch let err` instead of `catch { error = ... }`
- Prevention: Always use `catch let err` pattern

**Pattern 7: Type Mismatch in Date Formatting**
- Error: `Cannot convert value of type 'String' to expected argument type 'Date'`
- Fix: Parse ISO8601 string to `Date` first, then format
- Prevention: Check API response types (String vs Date)

**Pattern 8: Optional Binding on Non-Optional Values**
- Error: `Initializer for conditional binding must have Optional type`
- Fix: Check if value is actually optional, use index check instead
- Prevention: Verify type before using optional binding

**Pattern 9: Type Not in Scope (Nested Types)**
- Error: `Cannot find type 'NestedType' in scope`
- Fix: Use fully qualified name: `OuterType.NestedType`
- Prevention: Check type definition location

**Pattern 10: Wrong Property Name on ObservableObject**
- Error: `Value of type 'Type' has no dynamic member 'propertyName'`
- Fix: Verify actual property name in ViewModel class
- Prevention: Use `view` tool to read ViewModel definition

**Pattern 11: SwiftUI Type-Checking Timeout**
- Error: `The compiler is unable to type-check this expression in reasonable time`
- Fix: Break up complex `body` into smaller `@ViewBuilder` computed properties
- Pattern:
  ```swift
  var body: some View {
      NavigationStack {
          mainContent.toolbar { toolbarContent }
      }
  }
  
  @ViewBuilder
  private var mainContent: some View { ... }
  
  @ToolbarContentBuilder
  private var toolbarContent: some ToolbarContent { ... }
  ```

**Pattern 12: CLLocationCoordinate2D onChange Error**
- Error: `Referencing instance method 'onChange(of:initial:_:)' requires that 'CLLocationCoordinate2D' conform to 'Equatable'`
- Fix: Add extension in `Models/SharedTypes.swift`:
  ```swift
  extension CLLocationCoordinate2D: Equatable {
      public static func == (lhs: CLLocationCoordinate2D, rhs: CLLocationCoordinate2D) -> Bool {
          lhs.latitude == rhs.latitude && lhs.longitude == rhs.longitude
      }
  }
  ```

**Pattern 13: Switch Must Be Exhaustive**
- Error: `switch must be exhaustive` or `add missing case: '.caseName'`
- Fix: Add all enum cases to switch statement, including any new cases added to enum
- Prevention: When adding new enum cases, search for all switch statements using that enum:
  ```bash
  grep -r "switch.*EnumName" ios/
  ```
- Example: `TimeFilter` enum has `.custom` case → all switches must include `.custom`
- Common locations: Filter logic in FeedView, MapView, CreateEventView

**Pattern 14: Optional Binding on Non-Optional**
- Error: `Initializer for conditional binding must have Optional type, not 'Type'`
- Fix: Check if value is actually optional. If not, use direct assignment or check for empty/nil differently
- Prevention: Verify return type before using `if let`:
  ```swift
  // ❌ Wrong: formatAddressForDisplay returns String, not String?
  if let address = formatAddressForDisplay(from: item) { ... }
  
  // ✅ Correct: Check if empty instead
  let address = formatAddressForDisplay(from: item)
  if !address.isEmpty { ... }
  ```
- Common mistake: Functions returning `String` vs `String?` - always check return type

**Pattern 15: Main Actor Isolation in Closures**
- Error: `main actor-isolated property 'propertyName' can not be mutated from a Sendable closure`
- Fix: Wrap closure body in `Task { @MainActor in ... }` to ensure UI updates happen on main actor
- Prevention: When mutating `@Published` or `@State` properties from async closures (completion handlers, callbacks), wrap in `Task { @MainActor in ... }`
- Example:
  ```swift
  // ❌ Wrong: Mutating from Sendable closure
  geocoder.geocodeAddressString(text) { placemarks, error in
      viewModel.cameraPosition = ... // Error: main actor isolation
  }
  
  // ✅ Correct: Wrap in Task with @MainActor
  geocoder.geocodeAddressString(text) { placemarks, error in
      Task { @MainActor in
          viewModel.cameraPosition = ... // OK: on main actor
      }
  }
  ```

**Pattern 16: DatePicker Parameter Order**
- Error: `argument 'in' must precede argument 'displayedComponents'`
- Fix: `in:` parameter must come before `displayedComponents:` in DatePicker initializer
- Prevention: Always use correct parameter order:
  ```swift
  // ❌ Wrong: displayedComponents before in
  DatePicker("End", selection: $date, displayedComponents: .date, in: startDate...)
  
  // ✅ Correct: in before displayedComponents
  DatePicker("End", selection: $date, in: startDate..., displayedComponents: .date)
  ```
- Common when: Using both date range (`in:`) and component selection (`displayedComponents:`)

**Pattern 17: Deprecated API Usage (iOS 26+)**
- Warning: `'placemark' was deprecated in iOS 26.0: Use location, address and addressRepresentations instead`
- Fix: For now, continue using `placemark` (still functional). New API structure differs from expected
- Prevention: When using deprecated APIs:
  - Check if replacement API is available and documented
  - If replacement is unclear or doesn't work as expected, use deprecated API with comment
  - Add TODO comment for future migration
- Note: Deprecation warnings are acceptable if API still works and replacement is not clearly documented
