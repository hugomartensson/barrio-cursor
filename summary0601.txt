PROJECT OVERVIEW
----------------
BarrioCursor is a hyperlocal events discovery platform consisting of:
- Backend: Node.js/TypeScript REST API with Express.js
- Database: PostgreSQL 14+ with PostGIS extension
- Authentication: Supabase Auth (JWT-based)
- Storage: Supabase Storage for media files
- Frontend: iOS app built with SwiftUI

Project Structure:
- /server - Backend API
- /ios/BarrioCursor - iOS SwiftUI application

2. ACTOR-BASED CONCURRENCY (iOS)

3. AUTHENTICATION FLOW

   - Key tables: users, events, media_items, likes, going
   - Events use PostGIS `Geometry` type for location storage
   - Geospatial queries use `ST_DWithin` and `ST_Distance` functions

5. API DESIGN
   - RESTful endpoints with consistent error responses
   - All protected routes use `requireAuth` middleware
   - Request validation using Zod schemas
   - Standardized error format: `{ error: { code, message, details? } }`
   - Standardized success format: `{ data: T }`

AUTHENTICATION IMPLEMENTATION
------------------------------

1. SIGNUP FLOW (`POST /api/auth/signup`)
   - Validates email, password (min 8 chars, uppercase, lowercase, number), name
   - Creates user in Supabase Auth
   - Syncs user to local PostgreSQL `users` table via `userSync.syncUser()`
   - Returns: `{ data: { user, token, refreshToken } }`
   - Handles email confirmation disabled scenario

2. LOGIN FLOW (`POST /api/auth/login`)
   - Validates email and password
   - Authenticates with Supabase
   - Syncs user to local database
   - Returns: `{ data: { user, token, refreshToken } }`

3. TOKEN REFRESH (`POST /api/auth/refresh`)
   - Accepts refresh token in request body
   - Uses `supabaseAdmin.auth.refreshSession()` to get new tokens
   - Returns new access token and refresh token
   - Used automatically by iOS app on 401 errors

4. AUTH MIDDLEWARE (`requireAuth`)
   - Extracts Bearer token from Authorization header
   - Verifies token using `supabaseAdmin.auth.getUser(token)`
   - Attaches user info to request: `req.user = { userId, email }`
   - Returns 401 if token missing, invalid, or expired
   - Async/await implementation for proper error handling

USER SYNCHRONIZATION
--------------------
- Problem: Supabase Auth users need to exist in local PostgreSQL for foreign keys
- Solution: `userSync.syncUser()` called after signup/login
- Creates or updates user record in `users` table
- Prevents foreign key constraint violations when creating events

USERS (Protected)
  GET /api/users/me
    Headers: Authorization: Bearer <token>
    Response: { data: { id, email } }
  
  GET /api/users/me/events
    Headers: Authorization: Bearer <token>
    Response: { data: Event[] }

    - Sorted by creation date (newest first)
    - Includes media, user info, and all event details

EVENTS (Protected)
  POST /api/events
    Headers: Authorization: Bearer <token>
    Body: {
      title, description, category,
      latitude, longitude,
      startTime (ISO8601), endTime? (ISO8601),
      media? (array of { url, type })
    }
    Response: { data: Event }
    Note: media is optional (defaults to empty array)
  
  GET /api/events/nearby?lat=<lat>&lng=<lng>&limit?=<limit>
    Headers: Authorization: Bearer <token>
    Response: { data: Event[] }
    - Uses PostGIS ST_DWithin to find events within 5km radius
    - Calculates distance using ST_Distance
    - Returns events sorted by distance
  
  GET /api/events/:id?lat?=<lat>&lng?=<lng>
    Headers: Authorization: Bearer <token>
    Response: { data: Event }
    - Includes distance if lat/lng provided

INTERACTIONS (Protected)
  POST /api/events/:id/like
    Headers: Authorization: Bearer <token>
    Response: { data: { liked, going, likesCount, goingCount } }
    - Toggles like status, updates counts
  
  POST /api/events/:id/going
    Headers: Authorization: Bearer <token>
    Response: { data: { liked, going, likesCount, goingCount } }
    - Toggles going status, updates counts

UPLOAD (Protected)
  POST /api/upload
    Headers: Authorization: Bearer <token>
    Body: { image: <base64>, contentType: "image/jpeg" }
    Response: { data: { url, type } }
    - Accepts base64-encoded images in JSON
    - Uploads to Supabase Storage using service key
    - Returns public URL
    - Validates file type and size

DATABASE SCHEMA
---------------

Users Table
  - id (UUID, primary key) - matches Supabase Auth user ID
  - email (string, unique)
  - name (string)
  - profilePictureUrl (string, nullable)
  - createdAt (timestamp)

Events Table
  - id (UUID, primary key)
  - userId (UUID, foreign key -> users.id)
  - title (string, max 200)
  - description (string, max 2000)
  - category (enum: food_drink, arts_culture, music, nightlife, sports_outdoors, community)
  - location (PostGIS Geometry) - stores lat/lng as point
  - latitude, longitude (computed fields for queries)
  - startTime (timestamp)
  - endTime (timestamp, nullable)
  - createdAt (timestamp)
  - likesCount, goingCount (integers, default 0)

MediaItems Table
  - id (UUID, primary key)
  - eventId (UUID, foreign key -> events.id)
  - url (string) - Supabase Storage public URL
  - type (enum: photo, video)
  - order (integer) - for sorting

Likes Table
  - userId (UUID, foreign key -> users.id)
  - eventId (UUID, foreign key -> events.id)
  - createdAt (timestamp)
  - Primary key: (userId, eventId)

Going Table
  - userId (UUID, foreign key -> users.id)
  - eventId (UUID, foreign key -> events.id)
  - createdAt (timestamp)
  - Primary key: (userId, eventId)

GEOSPATIAL QUERIES
------------------
- Uses PostGIS functions:
  - `ST_MakePoint(longitude, latitude)` - create point from coordinates
  - `ST_DWithin(geometry1, geometry2, distance_meters)` - find within radius
  - `ST_Distance(geometry1, geometry2)` - calculate distance
- Default search radius: 5km (5000 meters)
- Events sorted by distance (ascending)

MEDIA UPLOAD IMPLEMENTATION
---------------------------
1. Client sends base64-encoded image in JSON to `/api/upload`
2. Backend decodes base64 to Buffer
3. Validates MIME type (image/jpeg, image/png, etc.)
4. Validates file size (max 10MB for images)
5. Generates unique file path: `{userId}/{uuid}.{ext}`
6. Uploads to Supabase Storage using service key (full access)
7. Returns public URL: `{supabaseUrl}/storage/v1/object/public/{bucket}/{path}`
8. Client uses returned URLs when creating events

AUTHENTICATION FLOW
--------------------

1. APP LAUNCH
   - ContentView checks Keychain for stored token
   - If token exists, shows MainTabView
   - If no token, shows AuthView

2. SIGNUP/LOGIN
   - User enters email, password, name (signup)
   - AuthManager calls APIService.signup/login
   - On success, stores token + refreshToken + user in Keychain
   - Sets isAuthenticated = true
   - ContentView automatically switches to MainTabView

3. TOKEN REFRESH
   - APIService automatically detects 401 errors in performRequest()
   - Calls attemptTokenRefresh() which reads refresh token from Keychain
   - Calls /api/auth/refresh endpoint
   - Updates Keychain with new tokens
   - Retries original request with new access token
   - If refresh fails, clears Keychain tokens
   - Views no longer need to handle token refresh manually

4. LOGOUT
   - Clears all Keychain entries
   - Sets isAuthenticated = false
   - ContentView switches back to AuthView

KEY TECHNICAL DECISIONS & RATIONALE
================================================================================

1. WHY SUPABASE AUTH INSTEAD OF CUSTOM JWT?
   - Faster MVP development
   - Built-in user management
   - Email/password handling
   - Token refresh mechanism
   - Can migrate to custom JWT later if needed

2. WHY BASE64 IMAGE UPLOAD?
   - Avoids Express body parsing conflicts
   - Works with existing JSON middleware
   - Simpler than multipart/form-data
   - Trade-off: ~33% size increase, acceptable for MVP

3. WHY ACTOR FOR APISERVICE?
   - Thread-safe concurrent network requests
   - Prevents data races in async code
   - Swift concurrency best practice
   - Isolates network state

4. WHY USER SYNC TO LOCAL DB?
   - Foreign key constraints require user records
   - Events.userId references users.id
   - Supabase Auth users don't automatically exist in local DB
   - Sync on signup/login ensures referential integrity

5. WHY POSTGIS FOR LOCATION?
   - Efficient geospatial queries
   - ST_DWithin for radius searches
   - ST_Distance for distance calculations
   - Industry standard for location-based apps
   - Better than calculating distance in application code

6. WHY MEDIA OPTIONAL?
   - Reduces friction for event creation
   - Users can add events quickly
   - Can add photos later (future feature)
   - Better UX for MVP

7. WHY TOKEN REFRESH IN VIEW?
   - Simpler than middleware approach
   - View has access to AuthManager
   - Can show user-friendly error messages
   - Can retry specific operations
   - Future: Extract to APIService interceptor

8. WHY KEYCHAIN NOT USERDEFAULTS?
   - More secure for sensitive data
   - Tokens encrypted by iOS
   - Best practice for authentication tokens
   - UserDefaults is plaintext

TROUBLESHOOTING
================================================================================

COMMON ISSUES
-------------

1. "Token expired" errors
   - Solution: Log out and log back in to get new refresh token
   - Future: Automatic background refresh

2. "Could not connect to server" (iOS Simulator)
   - Check Mac IP address in AppConfig.swift
   - Ensure backend is running on that IP
   - Check firewall settings

3. Foreign key constraint violations
   - Ensure userSync.syncUser() is called after signup/login
   - Check that Supabase user ID matches local user ID

4. Image upload fails with 403
   - Check Supabase storage bucket policy
   - Ensure "Allow authenticated uploads" policy exists
   - Verify bucket name matches SUPABASE_STORAGE_BUCKET

5. Geospatial queries return no results
   - Verify PostGIS extension is installed: `CREATE EXTENSION postgis;`
   - Check that events have valid lat/lng coordinates
   - Verify ST_DWithin radius (default 5000m)

6. Build errors in Xcode
   - Ensure all Swift files are added to target
   - Check Info.plist is not in "Copy Bundle Resources"
   - Verify bridging header is empty (pure Swift project)
